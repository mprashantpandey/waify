1. System Model

Core Architecture

  • Multi-tenant SaaS with strict account isolation.
  • Tenant owner plus tenant team member model.
  • Platform super admin panel for operations and oversight.
  • WhatsApp Cloud API integration with Embedded Signup and connection management.
  • Queue-backed messaging, campaigns, sync, alerts, and diagnostics flows.
  • Inertia/Laravel application with tenant and admin surfaces sharing a common design system.

Main Business Objects

  • Account/Tenant: the main customer boundary.
  • User: owner, team member, or platform admin.
  • WhatsApp Connection: mapped Meta asset set for a tenant.
  • Conversation/Message: inbox operational objects.
  • Template: Meta message template with local sync state.
  • Campaign: queued outbound broadcast workflow.
  • Contact/List/Segment: audience and consent targeting layer.
  • Subscription/Invoice/Payment/Usage: billing and access layer.
  • Alert/Diagnostics bundle: operational observability layer.

2. User Types and Access Surfaces

User Type Main Surface Allowed Actions Important Restrictions
Public visitor Landing, pricing, auth entry points Review plans, create account, log in, contact support No tenant data access
Tenant owner Tenant app All tenant modules, plan changes, billing recovery, team, connections, templates, campaigns Still blocked by subscription gates where appropriate
Tenant team member Tenant app Operational modules based on assigned permissions and role Cannot renew, resume, top up, change plans, or perform owner-only billing actions
Platform super admin Platform panel Plans, pricing snapshots, tenant oversight, health, support, system alerts, impersonation/admin actions Must not cross tenant isolation via normal tenant-scoped actions
API consumer Developer/API surface Use scoped API keys and tenant outbound webhooks Bound by tenant-scoped key scopes and webhook configuration

Tenant Owner Mental Model

  • Chooses plan and completes onboarding.
  • Connects WhatsApp assets via Embedded Signup or manual recovery flows.
  • Operates inbox, templates, campaigns, contacts, support, billing.
  • Owns team invites and most tenant settings.

Team Member Mental Model

  • Primarily an operator.
  • Works inside inbox and related operational modules.
  • Can review billing state but not execute owner-only billing recovery.
  • Can be assignment target in inbox.

Platform Admin Mental Model

  • Runs the platform, not a tenant.
  • Uses health, alerts, pricing, billing, support, and admin diagnostic tooling.
  • Needs high-visibility auditability and recovery controls.

3.5 Message Diagnostics in Inbox

The inbox supports support-facing diagnostics on outbound messages. Current diagnostics include provider payload retention, message status progression, error visibility, and message-level drill-down. This is already useful for support and is still being tightened for consistency.
  • Outbound statuses include queued/accepted/sent/delivered/read/failed style rendering depending on the page.
  • Exact provider error payloads are retained when available.
  • Timeline information can include accepted, delivered, and read timestamps.
  • Conversation and template surfaces both use diagnostics concepts, but consistency work is ongoing.

6. Core End-to-End Flows

6.1 New Tenant Commercial Flow

  1. Visitor signs up.
  2. User verifies account and enters onboarding.
  3. User selects plan or enters billing-relevant onboarding stage.
  4. System creates/activates tenant context.
  5. User proceeds to app dashboard and setup surfaces.
  6. Subscription gating determines what modules are available versus recovery-only views.

6.2 WhatsApp Connection Setup Flow

  1. Tenant owner opens Connections.
  2. Owner starts Embedded Signup or manual connection recovery flow.
  3. Meta callback returns to platform.
  4. System performs token exchange / connection creation / metadata persistence.
  5. Cross-tenant asset ownership is checked.
  6. Connection is activated or marked with recoverable sync/setup issues.
  7. Owner can manually trigger health sync if needed.

6.3 Inbound Webhook to Inbox Flow

  1. Meta sends webhook payload.
  2. Webhook is validated, stored, and processed through queue-backed handlers.
  3. Inbound message creates/updates conversation and message records.
  4. Realtime events update inbox list/detail screens.
  5. Fallback stream polling repairs delayed or missed client state.
  6. Unread counters and activity summaries are recalculated or repaired as needed.

6.4 Agent Reply Flow

  1. Agent opens conversation.
  2. System determines if free-form reply is allowed or template is required.
  3. Pre-send validation checks connection state, access state, policy state, and payload validity.
  4. Message enters send pipeline.
  5. Provider status updates feed back into diagnostics and timeline rendering.
  6. If failed, exact error context is retained where available.

6.5 Template Lifecycle Flow

  1. Tenant creates or edits template locally.
  2. Template is submitted/synced to Meta.
  3. Local template record maps to remote template id/name/language/state.
  4. Webhook and sync jobs update status, rejection reason, and eligibility.
  5. Template can be used from inbox, campaigns, or dedicated send test screens if it is sendable.
  6. Recent send history provides support diagnostics and failure context.

6.6 Campaign Flow

  1. Tenant selects template and audience.
  2. System performs preflight checks: template eligibility, audience consent, connection state, subscription state.
  3. Campaign is queued for dispatch.
  4. Per-recipient sends flow through pipeline and status tracking.
  5. Failures roll into campaign diagnostics and usage/billing views.

6.7 Billing Recovery Flow

  1. Subscription becomes past due, expired, canceled grace, or payment-failed.
  2. Billing overview and related pages display state-specific recovery CTAs.
  3. Owner can renew, resume, update payment, or review plans depending on state.
  4. Non-owner can view context but cannot execute owner-only recovery actions.
  5. Payment/order callbacks update subscription status and access restoration state.

7. Billing and Subscription Flows

StateCurrent MeaningUser ExperienceAccess Effect
Trial active Tenant is inside allowed trial period Normal use with onboarding and billing awareness Paid modules may be available depending on current rules
Past due Payment recovery required Billing pages show recovery CTAs and failure context Access gating applies outside billing/profile/support-safe paths
Expired Billing cycle ended without active coverage Renewal-focused recovery Restricted access until successful recovery
Canceled grace Cancel applied but current period still valid User may still resume before end date Normal use until end date, then restricted
Renewed Payment succeeded and active period restored Billing pages should reflect settled recovery state Access should be restored
Billing logic has undergone repeated hardening. Current expectation is: historical billing records remain stable, recovery CTAs are owner-aware, payment failures preserve failure reason context, and duplicate callbacks should not re-run commercial state transitions.

8. Diagnostics, Alerts, and Support Flows

Diagnostics Surfaces

  • Webhook diagnostics and replay
  • Connection health and sync state
  • Message diagnostics in inbox
  • Template diagnostics in list/detail/send pages
  • Campaign failure summaries
  • Billing history and payment detail context
  • Platform system health and Meta readiness

Alerting Surfaces

  • Platform alert center
  • Tenant alert center
  • Webhook failure visibility
  • Queue failure visibility
  • Connection health warnings
  • Billing failure and recovery context

Support Bundle Pattern

  • Template surfaces now support diagnostics bundle copy/export for failed send context.
  • Bundles include template identity, sync state, last Meta error, latest failed send, timeline, payload, and message identifiers.
  • This pattern is useful for support escalation, debugging, and AI-agent-assisted troubleshooting.

10. Known Partial Areas

AreaCurrent StateWhy AI Agent Should Treat It Carefully
Inbox realtime and unread counters Hardening System has strong repair/fallback flows, but this remains a high-regression operational area.
Template sync edge cases Hardening Meta-side changes, header media requirements, and name conflicts require diagnostics-first debugging.
Billing recovery edge transitions Hardening Expired → renewed, failed callback duplicates, and access restoration must be treated as sensitive state transitions.
Meta API/version-sensitive metadata Hardening Fields can be nullable or missing depending on API version and webhook payload shape.
Consent localization and CSV safety Hardening AI agents should avoid assuming consent defaults; use explicit stored states and suppression flags.

AI-Agent Usage Guidance

  • Assume the platform is already broad and operational. Prefer polishing and hardening existing flows over inventing parallel ones.
  • Respect tenant isolation first. Almost every action is tenant-scoped unless explicitly in platform admin context.
  • Treat billing, connection ownership, and outbound send policy as high-risk state domains.
  • When diagnosing failures, first check diagnostics surfaces, payload retention, and latest status reconciliation instead of assuming missing data.
  • When modifying UI, keep owner-vs-non-owner behavior explicit, especially in billing.
  • When modifying inbox behavior, preserve stream fallback and repair mechanics.
  • When modifying template behavior, preserve sendability alignment across inbox, campaigns, and template test flows.
